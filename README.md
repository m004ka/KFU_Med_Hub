## 1. Введение

Настоящее Техническое задание (ТЗ) определяет требования к разработке интеграционной платформы-хаба (далее — Платформа, Хаб) для медицинской санчасти Казанского федерального университета (КФУ). Документ является основой для проектирования, разработки, тестирования и приёмки программного продукта.

### 1.1. Цель проекта

Создание централизованной цифровой платформы, которая объединяет медицинский персонал, преподавателей и студентов КФУ, интегрирует разрозненные источники медицинских данных, внедряет методы искусственного интеллекта в клиническую практику и поддерживает научно-исследовательскую деятельность.

### 1.2. Контекст и предпосылки

Современная медицинская санчасть университета сталкивается с рядом системных проблем:

- **Разрозненность данных:** медицинская информация хранится в разных системах без единой точки доступа и анализа.
- **Слабая интеграция с образовательным процессом:** у студентов и преподавателей ограниченный доступ к реальным медицинским данным для исследований.
- **Отсутствие инструментов ИИ в клинической практике:** диагностика и анализ данных выполняются вручную без поддержки интеллектуальных систем.
- **Низкий уровень цифровой зрелости:** по оценке 2024 года — менее 40%, тогда как целевой показатель к 2030 году составляет не менее 80% согласно требованиям Минздрава РФ.

Платформа-хаб призвана решить эти проблемы путём создания единого цифрового пространства на базе отечественных технологий и собственных разработок КФУ — по аналогии с ведущими мировыми практиками построения медицинских data-платформ.

## 2. Назначение и границы системы

### 2.1. Назначение продукта

Платформа-хаб предназначена для:

- централизованного хранения и управления медицинскими датасетами;
- предоставления инструментов ИИ-анализа медицинских данных без необходимости программирования;
- поддержки коллаборативных исследовательских проектов между врачами, преподавателями и студентами;
- безопасной интеграции с республиканскими и федеральными медицинскими информационными системами;
- создания основы для дальнейшего развития «умной клиники» КФУ.

### 2.2. Границы системы

**В область разработки входит:**

- Веб-приложение (frontend + backend) с ролевым управлением доступом
- Интеграционные адаптеры для HL7 FHIR, DICOM, REST/SOAP
- Модуль каталога и управления датасетами
- Модуль запуска и мониторинга ИИ-задач
- Модуль управления задачами и исследовательскими проектами
- API для будущего подключения IoT-устройств и обучения ИИ-моделей

**Вне области разработки (в текущей версии):**

- Разработка собственных ИИ-моделей с нуля (только интеграция с существующими)
- Мобильное приложение (нативное iOS/Android)
- Обучение ИИ-моделей внутри платформы (архитектурная подготовка — да, реализация — нет)
- Замена существующих МИС — только интеграция с ними

## 3. Функциональные требования

### 3.1. Управление пользователями и доступом

Платформа должна поддерживать гибкую ролевую модель управления доступом (RBAC). Каждая роль обладает строго определённым набором разрешений:

| Функция | Администратор | Врач | Преподаватель | Студент | Исследователь |
|---------|:---:|:---:|:---:|:---:|:---:|
| Просмотр каталога датасетов | ✓ | ✓ | ✓ | ✓ | ✓ |
| Скачивание датасетов | ✓ | ✓ | ✓ | Ограничено | ✓ |
| Загрузка датасетов | ✓ | ✓ | ✗ | ✗ | ✓ |
| Запуск ИИ-анализа | ✓ | ✓ | ✓ | Ограничено | ✓ |
| Создание проектов/задач | ✓ | ✓ | ✓ | ✗ | ✓ |
| Участие в проектах | ✓ | ✓ | ✓ | ✓ | ✓ |
| Управление пользователями | ✓ | ✗ | ✗ | ✗ | ✗ |
| Доступ к системным настройкам | ✓ | ✗ | ✗ | ✗ | ✗ |
| Просмотр журналов аудита | ✓ | ✗ | ✗ | ✗ | Ограничено |

Ролевая модель должна поддерживать:

- единый вход (SSO) через корпоративную систему КФУ;
- двухфакторную аутентификацию (2FA) для ролей «Врач» и «Администратор»;
- делегирование и временное расширение прав с полным аудит-логированием;
- автоматическую блокировку учётных записей при подозрительной активности.

### 3.2. Каталог датасетов

Модуль каталога датасетов является центральным репозиторием медицинских данных платформы. Он должен обеспечивать:

- Загрузку датасетов в форматах: CSV, JSON, XLSX, DICOM, NIFTI, HL7 FHIR Bundle, XML, Parquet.
- Метаданные каждого датасета: название, описание, теги, источник, дата создания, лицензия, размер, количество записей, схема данных.
- Полнотекстовый поиск и фильтрацию по: типу данных, тегам, источнику, дате загрузки, владельцу.
- Предварительный просмотр (превью): первые 50 строк таблицы, статистические сводки по числовым полям, распределения.
- Версионирование датасетов с историей изменений.
- Гранулярное управление доступом: публичный / только для группы / приватный.
- Скачивание в исходном формате или с конвертацией (CSV → XLSX и т.д.).
- Автоматическую проверку качества данных при загрузке (дубли, пустые значения, выбросы).

### 3.3. Модуль ИИ-анализа

Платформа предоставляет пользователям интерфейс для запуска задач ИИ-анализа без необходимости программирования. Поддерживаемые сценарии:

| Задача ИИ-анализа | Тип модели | Фреймворк | Описание применения |
|-------------------|------------|-----------|---------------------|
| Классификация болезней по МКБ-11 | Мультикласс. классификатор | Scikit-learn / PyTorch | Автоматическая классификация диагнозов по тексту истории болезни в соответствии с МКБ-11 (ICD-11 WHO) |
| Анализ рентгеновских снимков | CNN (ResNet, EfficientNet) | TensorFlow / ONNX | Детекция патологий лёгких, костей; предварительный скрининг; передача через DICOM |
| NLP историй болезней | Трансформер (BERT-based) | Hugging Face | Извлечение клинических сущностей, суммаризация выписок, поиск по свободному тексту |
| Анализ лабораторных данных | Регрессия / аномалия | Scikit-learn | Выявление отклонений в результатах анализов, трендовый анализ показателей здоровья |
| Предиктивная аналитика рисков | Градиентный бустинг | PyTorch / XGBoost | Оценка риска осложнений, ранняя диагностика хронических заболеваний |
| Обработка медицинских изображений (МРТ, КТ) | Сегментация (U-Net) | PyTorch / ONNX | Сегментация органов и опухолей; поддержка планирования лечения |

Технические требования к модулю ИИ:

- Асинхронная очередь задач с отображением статуса в реальном времени (WebSocket/SSE).
- Хранение истории запусков и результатов для каждого пользователя.
- Возможность сравнения результатов разных прогонов.
- Экспорт результатов анализа в PDF, CSV, JSON.
- API-эндпоинты для программного запуска анализа (для интеграции с внешними системами).
- Архитектурная поддержка: отдельный микросервис AI Gateway, изолированное вычислительное окружение (GPU-ноды), возможность добавления новых моделей без перезапуска платформы.

### 3.4. Управление задачами и исследовательскими проектами

Платформа должна включать полноценный инструмент для коллаборации, обеспечивающий:

- Публикацию задач от медицинского персонала: описание проблемы, требуемые компетенции, дедлайн, ожидаемый результат.
- Создание исследовательских проектов с привлечением участников из разных ролей (врач + преподаватель + студент).
- Kanban-доску для управления задачами внутри проекта.
- Файловое хранилище в рамках проекта с историей версий.
- Систему уведомлений (in-app + email) о новых задачах, изменениях статуса, комментариях.
- Привязку датасетов и результатов ИИ-анализа к конкретным проектам.
- Публичный каталог открытых задач, доступный всем пользователям платформы.

### 3.5. Интеграция с внешними системами

| Протокол / Стандарт | Назначение | Приоритет | Описание |
|---------------------|------------|-----------|----------|

### 5.2. Высокоуровневая архитектура

Платформа строится по принципу микросервисной архитектуры с единым API-шлюзом. Основные компоненты:

- **API Gateway** (Spring Cloud Gateway) — единая точка входа, аутентификация, маршрутизация.
- **Auth Service** — управление пользователями, ролями, токенами, SSO-интеграция.
- **Dataset Service** — каталог, метаданные, загрузка/скачивание датасетов.
- **AI Gateway** — управление очередью ИИ-задач, вызов внешних моделей, хранение результатов.
- **Integration Service** — адаптеры HL7 FHIR, DICOM, REST/SOAP для внешних ИС.
- **Project Service** — управление задачами, проектами, уведомлениями.
- **Notification Service** — email и in-app уведомления (Kafka-consumer).
- **Audit Service** — централизованное логирование всех действий пользователей.

Взаимодействие между сервисами — через Apache Kafka (асинхронные события) и REST/gRPC (синхронные вызовы). Все сервисы работают в изолированных Docker-контейнерах, оркестрируемых Kubernetes.

### 5.3. Требования к безопасности данных

- Все персональные медицинские данные (ПМД) обрабатываются в соответствии с требованиями 152-ФЗ «О персональных данных».
- Чувствительные поля (ФИО, СНИЛС, диагнозы) хранятся в зашифрованном виде в БД.
- Передача данных — исключительно по TLS 1.3.
- Регулярный автоматический бэкап БД и объектного хранилища (ежедневно, с хранением 30 дней).
- Возможность развёртывания в закрытом контуре (без доступа к интернету) для работы с критическими данными.

## 7. Целевые показатели цифровой зрелости

Целевой уровень цифровой зрелости медицинской санчасти КФУ — не менее 80% к 2030 году. Оценка производится по методике Минздрава РФ.

| Год | Ключевые достижения | Целевой уровень зрелости | Метрики |
|-----|---------------------|--------------------------|---------|
| 2025 | Запуск MVP: ядро платформы, каталог датасетов, базовая аутентификация | 40% | ≥100 пользователей, ≥5 датасетов в каталоге |
| 2026 | Интеграция ИИ-модулей, подключение к республиканским ИС (FHIR, DICOM) | 55% | ≥3 работающих ИИ-сценария, интеграция с ≥2 внешними ИС |
| 2027 | Модуль коллаборации, предиктивная аналитика, расширение каталога | 65% | ≥10 активных исследовательских проектов, ≥20 датасетов |
| 2028 | Масштабирование, IoT-подключения, обучение модели внутри платформы (v1) | 75% | ≥500 MAU, ≥5 обученных внутренних моделей |
| 2029–2030 | Полная цифровизация процессов санчасти, интеграция всех источников данных | ≥80% | Соответствие критериям Минздрава по цифровой зрелости |

**Ключевые KPI платформы по итогам внедрения:**

- Сокращение времени на поиск и подготовку медицинских данных для исследований — на **60%**.
- Увеличение количества совместных исследовательских проектов (врачи + студенты + преподаватели) — в **5 раз**.
- Доля диагнозов, верифицированных с применением ИИ-инструментов, — не менее **30%** к 2028 году.
- Количество активных ИИ-сценариев в продуктивной эксплуатации — не менее **6** к 2027 году.
